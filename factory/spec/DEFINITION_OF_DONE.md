# DEFINITION_OF_DONE.md — Критерии готовности к запуску

## Что значит "продукт запущен"?

Продукт считается **запущенным** только когда выполнены **все** критерии ниже. Нет исключений.

---

## Обязательные критерии (для всех продуктов)

### 1. ✅ Landing Page

**Что должно быть:**
- Заголовок: что это и главная выгода (1 строка)
- Описание: кому это нужно и почему (2-3 строки)
- **Цена явно указана** (крупным шрифтом)
- CTA кнопка: "Купить за X ₽" или "Оформить подписку"
- Контактная информация (email, Telegram)
- Юридическая информация (ИП/ООО, ИНН, ОГРН)

**Проверка:**
- Страница доступна по URL (production)
- Открывается на mobile и desktop
- Загружается за <3 секунд
- Все ссылки работают

---

### 2. ✅ Payment Integration

**Что должно быть:**
- Интеграция с платёжной системой (Stripe, ЮKassa, Lemon Squeezy)
- Кнопка "Купить" ведёт на реальную форму оплаты
- После оплаты:
  - Пользователь получает доступ к продукту
  - Или получает email с инструкциями
  - Или перенаправляется на страницу "Спасибо"

**Проверка:**
- Тестовая оплата проходит успешно
- Реальная оплата работает (проверить на минимальной сумме)
- Пользователь получает то, за что заплатил

---

### 3. ✅ Analytics

**Что должно быть:**
- Tracking событий:
  - `page_view` — просмотр landing page
  - `button_click` — клик на CTA кнопку
  - `purchase_initiated` — начало оплаты
  - `purchase_completed` — успешная оплата
- Система аналитики настроена (Vercel Analytics, Google Analytics, или Plausible)

**Проверка:**
- События попадают в аналитику
- Можно увидеть воронку: просмотр → клик → оплата

---

### 4. ✅ MVP Functionality

**Что должно быть:**
- Базовая функциональность работает
- Пользователь может выполнить главную задачу продукта
- Нет критических багов
- UI понятен без инструкции

**Проверка:**
- Пройти пользовательский сценарий от начала до конца
- Проверить на mobile и desktop
- Попросить 1-2 человек протестировать (друзья, коллеги)

**Примеры MVP:**
- **QuickPoll**: создать опрос, получить ссылку, увидеть результаты
- **InstantCV**: загрузить данные, получить PDF резюме
- **MoodTracker**: добавить запись настроения, увидеть график за неделю

---

### 5. ✅ Production Deployment

**Что должно быть:**
- Продукт задеплоен на production (Vercel, Netlify, или аналог)
- URL продукта работает (не localhost)
- HTTPS работает (зелёный замок)
- Нет ошибок 500/404 на главных страницах

**Проверка:**
- Открыть URL в браузере (incognito mode)
- Проверить на mobile и desktop
- Проверить, что нет консольных ошибок (F12 → Console)

---

### 6. ✅ Risk Assessment

**Что должно быть:**
- Файл `ventures/V-YYYY-NNN-slug/RISKS.md` создан
- Описаны ключевые риски (3-5 пунктов):
  - Риск 1: описание + митигация
  - Риск 2: описание + митигация
  - ...
- Оценка времени до "kill" решения (если нет транзакций)

**Проверка:**
- Файл существует и заполнен
- Риски реалистичны (не "что если метеорит упадёт")
- Есть план действий для каждого риска

**Пример:**
```markdown
# RISKS.md

## Риск 1: Нет спроса
**Описание**: Никто не покупает за 499 ₽/месяц
**Митигация**:
- Снизить цену до 199 ₽
- Добавить бесплатный trial на 7 дней
- Искать другую ЦА (например, малый бизнес вместо блогеров)
**Дедлайн**: 14 дней без транзакций = kill

## Риск 2: Конкуренты
**Описание**: Есть бесплатный аналог с большей функциональностью
**Митигация**:
- Позиционировать как "простое решение за 5 минут"
- Добавить уникальную фичу (интеграция с X)
```

## Дополнительные критерии (опционально)

### 7. ⭕ Email Notifications (желательно)
- Welcome email после регистрации
- Подтверждение оплаты
- Инструкции по использованию

### 8. ⭕ FAQ Section (желательно)
- 3-5 самых частых вопросов на landing page

### 9. ⭕ Social Proof (желательно для LONG track)
- Отзывы пользователей
- Количество пользователей/продаж
- Упоминания в СМИ/соцсетях

---

## Timeline для проверки

### FAST Track (≤7 дней)
Проверка DoD **на 6-й день** после старта:
- Если все критерии выполнены → запуск
- Если 1-2 критерия не выполнены → fix за 24 часа
- Если >2 критериев не выполнены → пересмотр сроков или kill

### LONG Track (≤3 месяца)
Проверка DoD **на неделе 2-3** после старта:
- Все критерии должны быть выполнены
- Дополнительные критерии (7-9) — желательны

## Чеклист перед запуском

Используй этот чеклист для каждого продукта:

```markdown
## Definition of Done: [V-YYYY-NNN-slug]

- [ ] Landing Page доступна по URL
- [ ] Цена явно указана
- [ ] Payment integration работает (тестовая оплата прошла)
- [ ] Analytics настроена (события tracked)
- [ ] MVP функциональность работает (протестировано)
- [ ] Production deployment готов (HTTPS, нет ошибок)
- [ ] Risk assessment создан (RISKS.md)
- [ ] Юридическая информация указана (ИП/ООО, ИНН)

### Опционально
- [ ] Email notifications настроены
- [ ] FAQ секция добавлена
- [ ] Social proof добавлен

---

**Дата готовности**: YYYY-MM-DD
**Ответственный**: [Имя]
**Статус**: ✅ Готово / ⏳ В процессе / ❌ Не готово
```

---

## Что делать, если DoD не выполнен?

### Если 1 критерий не выполнен:
- Fix за 24 часа (FAST) или 3 дня (LONG)
- Приоритет #1 для команды

### Если 2+ критериев не выполнены:
- Пересмотреть сроки запуска
- Оценить: стоит ли продолжать?
- Возможно, проблема в планировании

### Если критерии не выполняются постоянно:
- Это сигнал, что процесс сломан
- Нужно пересмотреть workflow или team capacity

---

**Главное правило**: DoD — это не бюрократия, это защита от "запуска" полу-готовых продуктов, которые никто не купит.
